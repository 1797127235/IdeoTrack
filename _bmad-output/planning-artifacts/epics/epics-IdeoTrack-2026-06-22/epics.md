# 思政打卡 App — Epic 与 Story 拆分

## 概述

本文档将 PRD 中的 31 条功能需求、架构脊柱中的 17 条架构决策以及 UX 体验设计规范，拆分为 12 个 Epic 和 44 个可交付的 User Story。每个 Story 均使用标准的用户故事格式（作为……我想要……以便……）和 Gherkin 风格的验收标准（Given/When/Then/And）。

> **范围**：V1 MVP，覆盖学生（微信小程序）、辅导员（微信小程序）、管理员（Expo App）三端；管理员 Web 后台推迟到 V2。
> **分端策略**（AD-17）：学生端和辅导员端为微信小程序（`miniprogram/`），管理员端 V1 为 Expo App（`mobile/`），V2 补充 Web 后台。
> **语言**：本文件及所有下游文档使用中文。

## Epic 1：账号认证与权限体系

**目标**：建立安全、可靠的三角色登录与权限体系，确保学生、辅导员、管理员只能访问各自授权的功能与数据。

**覆盖需求**：FR-1、FR-2、AD-4、AD-5、AD-14、AD-15、NFR-4、NFR-6、NFR-7、UX-15。

### Story 1.1：学号/工号密码登录

作为一名学生/辅导员/管理员，
我想要使用学号/工号和密码登录 App，
以便进入对应角色的首页并使用功能。

**验收标准：**

- **Given** 用户已安装并打开 App
- **When** 用户在登录页输入正确的学号/工号和密码并点击登录
- **Then** 系统校验成功后返回对应角色的 JWT 令牌并跳转到角色首页
- **And** 登录失败时显示清晰的中文错误提示（如「账号或密码错误」）
- **And** 登录请求使用 HTTPS 传输，密码不在客户端持久化存储

### Story 1.2：首次登录强制修改初始密码

作为一名新用户，
我想要在首次登录后修改初始密码，
以便保障账号安全。

**验收标准：**

- **Given** 用户首次使用初始密码登录成功
- **When** 系统检测到 `is_initial_password` 标志为 true
- **Then** 强制跳转到修改密码页，新密码长度不少于 6 位且不能与初始密码相同
- **And** 修改成功后更新密码哈希并清除首次登录标志
- **And** 新密码使用 bcrypt 等算法加密存储（NFR-4）

### Story 1.3：账号锁定策略

作为一名系统用户，
我想要在连续输入错误密码后账号被临时锁定，
以便防止暴力破解。

**验收标准：**

- **Given** 用户在登录页连续 5 次输入错误密码
- **When** 第 5 次登录失败
- **Then** 系统锁定该账号 15 分钟，期间任何登录尝试均返回「账号已锁定，请 15 分钟后重试」
- **And** 锁定计时从最后一次失败尝试开始计算
- **And** 锁定状态在服务端维护，避免客户端绕过

### Story 1.4：角色与权限控制

作为一名系统用户，
我想要系统根据我的角色控制功能访问范围，
以便不同角色只看到该有的入口和数据。

**验收标准：**

- **Given** 用户已成功登录并获取 JWT
- **When** 用户访问受保护页面或 API
- **Then** 系统根据 JWT 中的 `role` 字段校验权限
- **And** 学生只能访问任务、打卡、日历、排行、个人中心（FR-2）
- **And** 辅导员只能访问所带班级的看板、复核、个人中心（FR-2、AD-5）
- **And** 管理员可以访问任务发布、组织管理、全校报告、名言库（FR-2）
- **And** 一个用户只能拥有一个角色（V1 不支持一人多角色）

### Story 1.5：JWT 令牌管理与安全存储

作为一名已登录用户，
我想要 App 安全地保存和自动携带登录令牌，
以便在有效期内无需重复登录。

**验收标准：**

- **Given** 用户登录成功
- **When** 系统返回 JWT 后
- **Then** 移动端将令牌存储在安全存储中（如 Expo SecureStore / Keychain / Keystore）
- **And** 每次受保护请求自动在 `Authorization: Bearer <token>` 头中携带令牌
- **And** 令牌过期后自动跳转到登录页，或按策略刷新（V1 可简化为重新登录）
- **And** 令牌中仅携带 `userId`、`role`、`exp` 等必要声明（AD-4）

### Story 1.6：学生微信登录与学号绑定

作为一名学生，
我想要在微信小程序里用微信一键登录并绑定我的学号，
以便免密快速进入应用。

**验收标准：**

- **Given** 学生首次打开小程序，未登录状态
- **When** 点击「微信登录」按钮
- **Then** 系统调用 `wx.login` 获取临时 code，发送给后端
- **And** 后端用 code 换取 openid，发现该 openid 未绑定任何学号
- **And** 跳转至「绑定学号」页面，学生输入学号 + 密码
- **And** 验证通过后，系统将 openid 写入 users 表的 `wechat_openid` 字段
- **And** 签发 JWT 返回给小程序，进入学生首页

- **Given** 学生已绑定学号，再次打开小程序
- **When** `wx.login` 获取 code 发给后端
- **Then** 后端识别已绑定的 openid，直接签发 JWT，无需再次输密码

- **Given** 学生换手机/换微信号登录
- **When** 用新微信登录并绑定同一学号
- **Then** 旧 openid 自动解绑，新 openid 生效（防止账号盗用）

**覆盖需求：** FR-1（学生登录部分）、AD-4、AD-17
**归属端：** 学生小程序（前端 Story 12.2，后端接口 Story 12.3）

## Epic 2：每日名言与首页体验

**目标**：为学生首页营造打卡仪式感，通过管理员可配置的名言库按日轮换展示每日名言。

**覆盖需求**：FR-3、FR-4、UX-7、UX-15、UX-16。

### Story 2.1：每日名言展示

作为一名学生，
我想要打开 App 首页时看到一句每日名言，
以便获得正向鼓励并开始打卡。

**验收标准：**

- **Given** 学生进入首页
- **When** 首页加载完成
- **Then** 展示当日的名言内容、出处（如有）
- **And** 同一自然日内同一用户看到的名言保持一致
- **And** 名言每日 00:00 更新，或首次打开时更新
- **And** 名言内容来自管理员配置的名言库中已启用条目

### Story 2.2：近 7 日名言滑动查看

作为一名学生，
我想要在首页左右滑动查看近 7 日的历史名言，
以便回顾之前的激励内容。

**验收标准：**

- **Given** 学生在首页名言区域
- **When** 用户左右滑动
- **Then** 展示前 6 天/后 6 天（共 7 天窗口）的名言记录
- **And** 滑动动画流畅，过渡时间符合 UX 转场规范
- **And** 未来日期的名言不可查看

### Story 2.3：名言库管理

作为一名管理员，
我想要在后台管理每日名言库，
以便控制首页展示内容。

**验收标准：**

- **Given** 管理员进入组织管理模块的名言库页面
- **When** 管理员添加、编辑、删除或启用/禁用名言
- **Then** 系统保存修改并即时生效
- **And** 单条名言文本不超过 200 字
- **And** 启用名言数量不足时，系统循环使用已启用名言
- **And** 系统根据启用状态按日轮换展示每日名言

我想要为签到配置地理围栏（学校/教学楼/班级范围），

- **When** 学生在任务详情页点击「立即打卡」
- **Then** 后端获取位置后判定是否命中适用围栏
- **And** 命中围栏：正常创建/更新打卡记录（复用 Story 4.1 逻辑）
- **And** 未命中任一适用围栏：拒绝打卡，返回 `CHECKIN_OUTSIDE_GEOFENCE`
- **And** 小程序展示「当前不在签到范围内，请到指定地点打卡」（UX-13 文字+图标）

- **And** 围栏坐标统一 gcj02 体系
- **And** 仅管理员可配置围栏，辅导员只读
- **And** 在 Web 配置 UI 就绪前，支持迁移脚本/种子 SQL 配置初始围栏

**覆盖需求：** FR-9（围栏扩展）、AD-1、AD-5、AD-14、AD-20
**归属端：** 后端（`api/src/domains/checkins`）+ 学生小程序（错误态）+ 管理员 Web（配置 UI，随 Epic 14.2）
**依赖：** Story 4.1（位置基础）、Epic 14.2（Web 配置 UI）、Epic 9.x（管理员功能基座）

**Dev Notes（地图服务使用边界）：**

- **围栏校验不依赖任何地图服务**。判定"点是否在围栏内"是纯几何问题：Haversine 公式算两点球面距离 ≤ `radius_meters` 即命中，零 API 调用、零第三方依赖、零延迟、零合规风险（学生位置不外发）。保持 AD-20 的 Haversine 方案，不要把校验链路绑到腾讯/高德等地图服务上。
- **地图服务仅用于管理员配置体验**。Web 配置页（随 Epic 14.2）建议接入腾讯地图或高德地图的 JS SDK「选点组件」，让管理员在地图上点选/搜索地点自动填入经纬度 + 拖拽设半径，而非手敲 `116.397, 39.908`。这是前端体验优化，非架构决策。
- 分层边界：`[配置时] Web 选点组件 → 填坐标 → 存 geofences 表`；`[打卡时] 小程序 gcj02 → 后端 Haversine → 判定`。打卡链路零地图服务调用。
- **坐标系一致性**：选点组件、`wx.getLocation`、`geofences` 存储统一使用 gcj02，避免国测局/WGS84 混用导致偏移。配置选点组件时确认其返回 gcj02。
- **地址展示（可选）**：学生端展示「当前不在签到范围内」即可，无需逆地理编码；若需展示具体地址，优先用 `wx.getLocation` 自带 address，避免额外地图 API 调用。
- **选点组件候选**：腾讯地图（与微信生态同源，gcj02，与小程序一致，推荐）、高德地图（需注意坐标系转换）。两者均有成熟的 Web 选点组件，按团队熟悉度二选一即可。

## Epic 5：心得审核流程

**目标**：通过 AI 初审快速过滤敷衍内容和敏感词，未通过的内容进入辅导员人工复核，确保打卡质量。

**覆盖需求**：FR-12、FR-13、FR-14、AD-3、AD-5、AD-6、AD-10、AD-11、AD-12、NFR-8、UX-15。

### Story 5.1：AI 初审服务

作为一名系统，
我想要自动对学生提交的心得进行初步审核，
以便快速判断内容是否敷衍或包含敏感词。

- **Given** 学生提交心得
- **When** 系统调用 AI 初审服务
- **Then** 检测字数：少于 10 字判定为敷衍

- **And** 检测敏感词：命中 V1 预设敏感词库判定为需复核
- **And** AI 初审结果在 3 秒内返回（NFR-2）
- **And** AI 初审记录保存，便于后续调优和审计（NFR-8）

- **Then** 列表展示学生姓名、任务、心得内容、AI 初审结果
- **And** 辅导员可以选择「通过」「不通过」或「要求修改」

以便支撑小程序的学生登录流程。

**验收标准：**

- **Given** `api/src/domains/auth`
- **When** 实现 `POST /api/auth/wechat/login`
- **Then** 接收 code，调用微信 `code2session` 接口换取 openid
- **And** 若 openid 已绑定用户，签发 JWT 返回
- **And** 若未绑定，返回 `{ needBind: true }`

- **Given** 实现 `POST /api/auth/wechat/bind`
- **When** 接收 openid + 学号 + 密码
- **Then** 验证学号密码，绑定 openid（旧 openid 自动解绑）
- **And** 签发 JWT 返回

- **And** `users` 表新增 `wechat_openid` 字段（nullable，唯一索引）
- **And** 微信 AppID 和 AppSecret 通过环境变量配置（不入库）

**覆盖需求：** FR-1、AD-4、AD-17
**归属端：** 后端（`api/`）

## Epic 13：运维可见性与数据保护

**目标**：保障 MVP 上线后的数据安全（每日备份）与运维可见性，让管理员无需登录服务器即可掌握系统健康、备份、资源与错误日志状态。

**覆盖需求**：NFR-8（日志审计）、NFR-10（每日备份）、新增 FR-32（运维仪表盘）、AD-18
**归属端**：后端（`api/src/domains/ops`）+ 管理员 Expo App（`mobile/`）+ 基础设施（`docker-compose` / cron）

**范围说明**：
- Story 13.1（每日备份）、13.2（日志轮转落盘）为 **MVP 必做**，对应 NFR-10/NFR-8（此前 PRD 已要求但无 story 承接，本 Epic 闭合该漏洞）。
- Story 13.3（运维 API）、13.4（仪表盘）为 **MVP 前增强**，不阻塞核心打卡链路。

### Story 13.1：数据库每日备份（MVP 必做）

作为一名项目维护者，
我想要系统每日自动备份数据库，
以便在数据丢失或损坏时能恢复。

**验收标准：**

- **Given** 服务器已部署
- **When** 到达每日凌晨 3:00（cron 触发）
- **Then** 执行 `pg_dump` 导出到 `/opt/IdeoTrack/backups/db-backup-YYYYMMDD-HHMMSS.sql`
- **And** 保留最近 14 天的备份，超出自动删除
- **And** 备份脚本幂等，可手动重跑
- **And** 备份不依赖后端进程（走系统 cron），API 挂了备份照跑
- **And** 异地备份（对象存储 / 另一台机器）留待 V2

**归属端**：基础设施（`scripts/backup-db.sh` + 服务器 cron）

### Story 13.2：日志轮转与生产落盘（MVP 必做）

作为一名项目维护者，
我想要生产日志持久化落盘并自动轮转，
以便排查问题时有历史日志且不会写满磁盘。

**验收标准：**

- **Given** 生产容器运行中
- **When** 日志产生时
- **Then** 三容器（postgres / api / caddy）均配置 `json-file` 日志驱动，`max-size 10m / max-file 3`
- **And** api 容器挂载 `./logs:/app/logs` 卷，并设 `LOG_FILE_DIR=/app/logs`
- **And** pino 同时写 stdout 和 `/app/logs/app.log`，落盘到宿主机持久化
- **And** 容器重建后宿主机日志文件不丢失

**归属端**：基础设施（`docker-compose.yml`）

### Story 13.3：后端运维 API（ops 域，admin-only）

作为一名管理员，
我想要通过 API 读取系统运维状态，
以便运维仪表盘展示数据。

**验收标准：**

- **Given** 新增 `api/src/domains/ops` 域
- **When** 实现以下端点（全部 `authenticate` + `requireRoles('admin')`）
- **Then** 提供：
  - `GET /api/ops/health` —— API 进程存活 + Postgres 连通性（`SELECT 1`）+ 运行时长
  - `GET /api/ops/backups` —— 列 `/app/backups` 下备份文件（文件名 / 大小 / 时间 / 是否成功）
  - `GET /api/ops/system` —— 磁盘使用率 + Node 进程内存 + 容器运行时长
  - `GET /api/ops/logs?limit=50&level=error` —— 读 `app.log` 尾部 N 行按 level 过滤
- **And** 所有路径硬编码到挂载点，不接受用户输入拼路径（防目录穿越）
- **And** fs / DB 读取失败时返回降级数据（null），不让运维页搞挂后端

**归属端**：后端（`api/src/domains/ops/`）

### Story 13.4：管理员运维仪表盘（App）

作为一名管理员，
我想要在 App 内查看系统运维状态，
以便不登录服务器也能掌握系统健康。

**验收标准：**

- **Given** 管理员进入「系统状态」页
- **When** 页面加载或下拉刷新
- **Then** 展示四区：
  1. 服务健康：API✅ / Postgres✅ + 运行时长
  2. 系统资源：磁盘进度条、内存占用
  3. 备份记录：最近备份列表（时间 / 大小 / 状态），最近一次距今多久高亮
  4. 错误日志：最近 N 条 error 日志（时间 + 消息），可展开详情
- **And** 管理员首页网格新增「🛠️ 系统状态」卡片入口
- **And** 不新增 Tab；样式复用 theme tokens

**归属端**：管理员 Expo App（`mobile/app/(admin)/system-status.tsx` + `mobile/services/systemOpsApi.ts`）

## 跨 Epic 通用实现任务

以下技术任务未单独列为 Story，但应在各 Epic 实施过程中统一落地：

1. **后端项目初始化**：Node.js + Express + TypeScript 项目骨架、统一响应封装、错误处理中间件、日志规范（AD-9、NFR-11、NFR-13）。
