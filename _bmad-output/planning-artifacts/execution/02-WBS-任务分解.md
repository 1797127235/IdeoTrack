---

## 1. 项目概况

**一句话：** 面向高校大学生的思政打卡移动端 App，支持学生每日任务打卡、AI 初审 + 辅导员人工复核、积分等级激励、班级排行榜，以及辅导员数据看板与管理员多维度报告导出。

## 4. 依赖关系图说明（关键路径）

```
[1 基础架构] ──→ [2 账号与权限]
                      │
                      ├────→ [3 首页与每日名言]
                      │
                      ├────→ [4 任务管理] ──→ [5 打卡流程]
                      │                         │
                      │                         ├────→ [6 心得审核]
                      │                         │           │
                      │                         │           ├────→ [7 激励机制]
                      │                         │           │
                      │                         │           └────→ [12 通知与提醒]
                      │                         │
                      │                         └────→ [8 班级排行榜]
                      │
                      ├────→ [9 辅导员数据看板] ──→ [6 心得审核]
                      │                              │
                      │                              └────→ [12 通知与提醒]
                      │
                      └────→ [10 管理员后台] ──→ [11 报告与统计]

[13 系统测试与部署] 依赖所有模块完成
```

**关键路径说明：**
1. **基础架构 → 账号与权限 → 所有业务模块：** 账号与权限是进入任何功能的入口，必须先完成。
2. **任务管理 → 打卡流程 → 心得审核 → 激励机制：** 这是学生核心打卡链路的正向依赖。没有任务就没有打卡，没有打卡就没有审核，没有审核通过就没有积分/勋章/日历更新。
3. **心得审核 → 辅导员数据看板：** 辅导员看板中的「待复核」数据依赖审核模块产生。
4. **管理员后台 → 报告与统计：** 组织结构管理是报告按学院/班级维度统计的前提。
5. **所有模块 → 系统测试与部署：** 测试与部署在全部业务功能完成后进行。

## 5. 风险与缓冲

| 风险项 | 涉及 WBS 编号 | 风险描述 | 应对建议 |
|-------|:-----------:|---------|---------|
| **AI 初审准确率不足** | 6.1.1, 6.1.2 | 敏感词库和敷衍规则在真实数据中可能误判或漏判，导致辅导员复核工作量不可控 | ① 预留 2 天缓冲用于规则调优；② 将敏感词和套话模板配置化（无需管理员界面，但后端可热更新）；③ 在测试阶段用真实样本文本测试 |
| **PDF 报告生成复杂度高** | 11.2.2 | 动态图表嵌入 PDF 对技术选型要求高（如 Puppeteer / 报表库），可能超工时 | ① 早期技术预研（WBS 1.1 中预留调研时间）；② 备选方案：先导出 Excel + 静态摘要页，PDF 图表采用预生成图片拼接；③ 预留 1–2 天缓冲 |
| **跨模块数据一致性** | 4.4.1, 5.3.2, 7.1.1, 8.2.1 | 任务状态、打卡结果、积分、排行榜数据由多个模块计算，易出现数据不一致 | ① 定义统一的状态机与事件驱动机制（如打卡完成事件触发积分更新和排行榜刷新）；② 核心计算逻辑集中在后端服务层，避免前端重复计算；③ 增加每日数据对账脚本（13.1.1） |
