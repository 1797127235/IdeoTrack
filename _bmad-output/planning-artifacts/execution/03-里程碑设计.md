# 思政打卡 App (IdeoTrack) — 里程碑设计

> 本文档基于《思政打卡 App PRD》及《产品简介》制定，定义 V1 MVP 的开发里程碑计划、验收标准与关键路径。

---

## 1. 项目概况与计划假设

### 1.1 项目概况

IdeoTrack 是一款面向高校大学生的移动端思政学习打卡工具，覆盖学生、辅导员、管理员三种角色。V1 MVP 目标是在可控工期内完成三条核心链路的端到端可用：
- **学生链路**：查看任务 → 定位签到 → 提交心得 → AI 初审 → 获得积分
- **辅导员链路**：查看班级数据 → 复核心得 → 导出数据
- **管理员链路**：发布任务 → 管理组织 → 查看报表 → 导出报告

### 1.2 计划假设

| 假设项 | 内容 |
|--------|------|
| 总工期 | 4–5 周（假设项目启动后第 1 周至第 5 周） |
| 团队规模 | 2–3 人（1 名全栈/前端 + 1 名后端 + 1 名设计/产品/测试兼职） |
| 技术栈 | 跨平台移动框架（Flutter / React Native）+ 后端 REST API + 关系型数据库 |
| 部署目标 | 单一学校本地部署或云服务器，Android 端为主要验证端 |
| 数据规模 | 演示数据集：学生 500 人以内、辅导员 10 人以内、管理员 2–3 人 |

### 1.3 计划约束

- 时间节点统一以 **Week X**（第 X 周）表示，不做具体日期假设。
- 各里程碑验收标准必须与 PRD §6 Success Metrics 对齐。
- 假设团队具备移动端开发、后端 API 开发及数据库设计的基础能力。
- 因团队规模有限，设计、测试、产品工作由前端或后端兼任，需合理分配时间。

### 3.1 M1：需求与设计冻结（Week 1）

#### 3.1.1 目标

在 Week 1 结束前，完成所有需求对齐、UI/UX 设计、数据库与 API 设计，确保开发团队对「做什么」和「怎么做」达成共识，消除后续返工风险。

#### 3.1.2 包含的主要工作包

| 工作包 | 内容描述 | 对标 PRD 功能模块 |
|--------|----------|-------------------|
| **需求对齐** | 团队通读 PRD 与 Brief，确认需求边界、排定优先级、记录技术疑问 | §4 全部、§5 MVP Scope |
| **UI/UX 设计** | 完成核心页面高保真设计稿：学生首页、任务详情、打卡页、个人中心、辅导员看板、管理员后台、报表页 | §4.1–§4.11 |
| **数据库设计** | 设计用户表、任务表、打卡记录表、心得表、审核记录表、积分/等级表、名言库表、组织结构表等 ER 图与 DDL | §4.1、§4.3、§4.4、§4.6、§4.9 |
| **API 接口设计** | 定义 RESTful API 规范：认证、任务、打卡、审核、统计、报表、通知等接口的 URL、请求/响应格式、状态码 | §4 全部 |
| **技术选型** | 确认前端框架（Flutter/React Native）、后端框架、数据库、部署方式、AI 初审实现方案（规则引擎或轻量 NLP） | §9 NFRs |
| **项目脚手架** | 搭建前后端项目骨架、代码仓库、CI/CD 流水线（可选）、开发环境统一 | §9.4 Maintainability |

#### 3.1.3 交付物清单

1. 产品需求确认单（团队签字/确认记录）
2. UI 设计稿（Figma / Sketch / 图片集）：覆盖全部核心页面
3. 数据库 ER 图及 DDL 脚本
4. API 接口文档（Swagger / Markdown）
5. 技术选型报告（含 AI 初审方案对比）
6. 项目代码仓库（初始化 + 基础目录结构）
7. 开发环境搭建指南（README）

#### 3.1.4 验收标准

| 编号 | 验收项 | 标准 | 对标 PRD |
|------|--------|------|----------|
| M1-AC1 | 需求一致性 | 设计方案与 PRD 需求编号（FR-1 ~ FR-28）覆盖率达到 100% | §5 MVP Scope |
| M1-AC2 | 设计稿完整性 | 设计稿覆盖学生端、辅导员端、管理员端全部核心页面，无空白页面 | §4 Features |
| M1-AC3 | 接口可评审性 | API 文档包含所有接口的 URL、Method、请求/响应示例、错误码 | §4 Features |
| M1-AC4 | 数据库可落地性 | ER 图包含所有必要实体、关系、主键/外键、索引设计 | §4.3–§4.6 |
| M1-AC5 | 技术方案可行性 | AI 初审方案在演示数据集上可运行，响应时间 ≤ 3 秒（非高峰期） | §9.1 Performance |

#### 3.1.5 风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 需求理解偏差 | 返工 | 每日 15 分钟站会对齐，M1 结束前召开设计评审会 |
| 技术选型失误 | 阻塞开发 | 预留 1 天技术预研（如 AI 初审原型验证），必要时降级为规则引擎 |
| 设计稿输出延迟 | 拖慢开发 | 优先完成高频页面（学生首页、打卡页、辅导员看板），其余页面可后续补全 |

### 3.2 M2：核心功能开发（Week 2–3）

#### 3.2.1 目标

在 Week 2–3 完成学生端、辅导员端、管理员端的核心业务逻辑，使「学生打卡」和「辅导员审核」两条主链路可完整运行。

#### 3.2.2 包含的主要工作包

| 工作包 | 内容描述 | 对标 PRD 功能模块 |
|--------|----------|-------------------|
| **账号与权限** | 实现学号/工号 + 密码登录、角色权限控制、首次登录强制改密码、密码加密存储 | §4.1 FR-1、FR-2、§9.2 |
| **首页与每日名言** | 实现学生首页名言展示、名言库配置、历史名言滑动 | §4.2 FR-3、FR-4 |
| **任务管理** | 实现任务发布（管理员/辅导员）、任务列表展示、任务详情、任务状态计算 | §4.3 FR-5 ~ FR-8 |
| **打卡流程** | 实现定位签到（仅记录经纬度）、心得提交（10–500 字）、打卡完成反馈 | §4.4 FR-9 ~ FR-11 |
| **AI 初审** | 实现字数检测、Jaccard 相似度检测、敏感词检测、结果返回 | §4.5 FR-12 |
| **人工复核** | 实现辅导员待复核列表、通过/不通过/要求修改操作、学生查看复核结果 | §4.5 FR-13、FR-14 |
| **激励机制** | 实现积分计算（基础 + 连续加成）、等级晋升、勋章授予、打卡日历展示 | §4.6 FR-15 ~ FR-18 |
| **组织结构** | 实现学院/班级增删改、用户批量导入 | §4.9 FR-25、FR-26 |

#### 3.2.3 交付物清单

1. 可运行的学生端 App（Android 验证端）
2. 可运行的辅导员端数据看板（嵌于 App 内）
3. 后端 REST API（覆盖 FR-1 ~ FR-18）
4. AI 初审模块（规则引擎或轻量 NLP）
5. 数据库种子数据（演示用户、任务、名言）
6. 单元测试覆盖核心业务逻辑（积分计算、打卡状态机、审核流程）

#### 3.2.4 验收标准

| 编号 | 验收项 | 标准 | 对标 PRD §6 |
|------|--------|------|-------------|
| M2-AC1 | 学生打卡完成率 | 演示数据集下，当日有效打卡学生数 / 应打卡学生数 ≥ 80% | SM-1 |
| M2-AC2 | 学生打卡操作步骤 | 学生从打开任务到完成打卡 ≤ 4 步（看任务 → 签到 → 写心得 → 提交） | SM-4 |
| M2-AC3 | AI 初审覆盖率 | AI 成功完成初审的心得数 / 总提交心得数 ≥ 99% | SM-5 |
| M2-AC4 | AI 初审响应时间 | 非高峰期接口响应 ≤ 3 秒 | §9.1 Performance |
| M2-AC5 | 密码安全 | 用户密码加密存储（bcrypt 或同等算法） | §9.2 Security |
| M2-AC6 | 角色隔离 | 学生/辅导员/管理员角色功能访问正确隔离 | §4.1 FR-2 |
| M2-AC7 | 连续打卡与积分 | 积分计算、等级晋升、勋章授予逻辑正确，单元测试通过 | §4.6 FR-15 ~ FR-17 |

#### 3.2.5 风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI 初审准确率不足 | 大量内容进入人工复核，辅导员负担重 | 预设保守规则（字数 + 敏感词优先），Jaccard 阈值可微调；M2 结束前调优 |
| 定位权限兼容性问题 | 部分机型无法签到 | 测试覆盖主流 Android 机型；拒绝定位时给出明确提示 |
| 并发打卡性能问题 | 高峰期卡顿 | 数据库索引优化；M2 预留 1 天性能测试 |
| 跨端逻辑差异 | 学生端与辅导员端代码复用度低 | 组件化设计，公共逻辑抽离为独立模块 |

### 3.3 M3：数据与报表集成（Week 3–4）

#### 3.3.1 目标

在 Week 3–4 完成辅导员数据看板、管理员统计报表、班级排行榜、通知系统，使「辅导员查看数据」和「管理员生成报告」两条链路完整可用。

#### 3.3.2 包含的主要工作包

| **辅导员数据看板** | 实现班级数据概览、未打卡学生名单（标红）、一键提醒、数据导出（Excel） | §4.8 FR-21 ~ FR-24 |

| **数据准确性校验** | 报表数据与原始打卡记录核对，确保一致性 | §4.10 |

#### 3.3.3 交付物清单

1. 完整的辅导员数据看板（含未打卡名单、一键提醒、数据导出）
2. 管理员统计报表页面（含图表、下钻、异常标注）
3. 报告导出功能（Excel + PDF）
4. 班级排行榜页面
5. 通知推送模块（App 内推送）
6. 数据核对报告（报表数据与原始数据一致性验证）

#### 3.3.4 验收标准

| 编号 | 验收项 | 标准 | 对标 PRD §6 |
|------|--------|------|-------------|
| M3-AC1 | 辅导员查看班级数据效率 | 辅导员从登录到查看班级打卡率的点击次数 ≤ 2 次 | SM-2 |
| M3-AC2 | 报告导出成功率 | 正常网络和数据规模下，导出成功次数 / 总尝试次数 ≥ 95%，无未处理异常 | SM-3 |
| M3-AC3 | 数据准确性 | 报表中的打卡率、未打卡名单与原始打卡记录 100% 一致 | Brief §6 |
| M3-AC4 | 报表查询性能 | 默认时间范围内数据查询 ≤ 5 秒 | §9.1 Performance |
| M3-AC5 | 人工复核效率 | 辅导员从收到待复核提醒到完成复核的平均时间 ≤ 5 分钟（演示场景） | SM-6 |
| M3-AC6 | 激励感知 | 学生能清晰看到积分、等级、连续打卡天数、班级排名 | Brief §6 |

#### 3.3.5 风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| PDF 报告生成复杂 | 延迟交付 | 先用基础模板生成 PDF（封面 + 表格），图表以静态图片嵌入；M3 前半周完成原型 |
| 报表数据延迟/不一致 | 决策错误 | 排行榜和看板数据允许 5 分钟延迟（PRD 定义）；定时任务 + 缓存策略 |
| 通知推送失败 | 提醒失效 | 本地通知兜底；推送失败记录日志 |
| 数据量大导致报表慢 | 体验差 | 默认时间范围限制（如本月）；数据库索引 + 预聚合 |

## 4. 关键路径说明

以下依赖关系定义了各里程碑之间的关键路径，任何前序节点的延迟都会直接影响后续节点：

1. **M1 → M2**：M1 的 UI 设计稿和 API 接口文档是 M2 开发的前置条件。数据库设计必须在 M2 编码开始前冻结。
2. **M2 → M3**：M2 完成的账号体系、打卡流程、AI 初审是 M3 辅导员看板和排行榜的数据来源。M3 的报表依赖 M2 产生的打卡记录和审核数据。
3. **M3 → M4**：M3 完成的全部功能模块是 M4 测试的对象。报告导出功能必须在 M4 测试前完成开发。
4. **M2 内部依赖**：
   - 账号与权限（FR-1/FR-2）必须先于其他所有功能完成，否则角色隔离无法验证。
   - 任务管理（FR-5 ~ FR-8）必须先于打卡流程（FR-9 ~ FR-11），否则无任务可打卡。
   - 打卡流程必须先于 AI 初审（FR-12）和人工复核（FR-13），否则无数据可审核。
5. **M3 内部依赖**：
   - 辅导员数据看板（FR-21 ~ FR-23）必须先于报告导出（FR-28），因为报告数据来源于看板统计逻辑。
   - 通知系统（FR-27 通知）依赖打卡和审核事件触发，需在打卡/审核完成后接入事件监听。
6. **M4 内部依赖**：
   - 冒烟测试优先于全量功能测试，确保核心链路通顺。
   - 缺陷修复优先于部署准备，避免带缺陷上线。

## 5. 资源分配建议

假设团队 2–3 人，建议按以下角色分工：

| 角色 | 人数 | 主要职责 | 各阶段工作分配 |
|------|------|----------|----------------|
| **前端/移动端开发** | 1 人 | 负责学生端、辅导员端、管理员端 UI 开发；兼任 UI 设计（如团队无专职设计） | M1：参与设计评审，输出基础组件；M2：学生端核心页面；M3：看板、报表、排行榜；M4：缺陷修复、界面优化 |
| **后端开发** | 1 人 | 负责 API 开发、数据库设计、AI 初审模块、权限控制、报表生成 | M1：数据库设计、API 设计；M2：核心业务逻辑、AI 初审；M3：统计报表、数据导出；M4：性能优化、安全加固 |
| **产品/设计/测试** | 1 人（兼职） | 负责需求确认、UI 设计（Figma）、测试用例编写、用户手册、答辩材料 | M1：UI 设计、需求确认；M2：测试用例准备；M3：功能测试、数据核对；M4：全量测试、手册、PPT |

### 补充说明

- 若团队仅有 2 人，建议「前端/移动端开发」兼任 UI 设计（使用现成 UI 组件库），「后端开发」兼任测试和部分产品工作。
- 每日建议保持 15 分钟站会，对齐进度和风险。
- M2 和 M3 是工作量最大的阶段，建议两人结对或并行开发：前端与后端按接口契约并行推进，减少阻塞。
- 建议在 M2 中期（Week 2 后半周）进行一次联调，确保前后端接口对接顺畅。
